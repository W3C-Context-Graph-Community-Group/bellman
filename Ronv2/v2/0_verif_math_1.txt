Operational Issue (input → operations → output) – Dark Fraction Theorem
Input:
m – number of shared variables at the boundary between two systems (integer, m ≥ 1; examples in the table: 5, 10, 20, 50)
r – number of verified facets (facet = Meaning / Structure / Context) through cross-boundary measurement (integer, 0 ≤ r ≤ n)
Variables and ranges:
n = 3 × m → total number of facets (each variable gives 3 bits: M, S, C)
examples: n = 15, 30, 60, 150
configuration space = {0, 1}^n (each bit is 0 = URLs match / 1 = they do not match)
Operations (step by step):
Oblicz n = 3 * m
Oblicz objętość kuli Hamminga (ball_volume) = suma od k=0 do r włącznie z (n choose k)
Oblicz rozmiar całej przestrzeni = 2^n
verified_fraction = ball_volume / 2^n
dark_fraction δ = 1 − verified_fraction
Output:
δ(n, r) – dark fraction as a number in the range [0, 1) – exactly how many configurations are unverifiable by in-system diagnostics
additionally: ϕ(n, r) = verified fraction (verified fraction)
for fixed r and increasing m → δ → 1 (corollary)
Does the mathematical / accounting calculation make sense?
Yes – the calculation is 100% correct and holds together.
The configuration space has exactly 2^n points – standard.
Verification of r facets covers exactly the Hamming ball of radius r around the zero vector (all zeros = full agreement) – the Hamming definition is native here for the protocol (only match/mismatch).
The number of points in the ball = sum of binomial(n, k) from k=0 to r – this is exactly the formula for the volume of a Hamming ball in the hypercube, with no simplifications.
